\# 配置管理方案



\## 一、配置项识别



根据实验要求，本项目识别以下主要配置项：



| 配置项 | 文件路径 | 说明 |

|--------|----------|------|

| 项目说明 | `README.md` | 项目概况、团队分工、协作约定 |

| WBS文档 | `docs/02-wbs.md` | 三级工作分解结构（交付物导向） |

| 进度计划 | `docs/03-schedule.md` | 简化进度计划（活动、工期、前置关系、责任人） |

| 配置管理文档 | `docs/04-configuration.md` | 本文件，记录配置管理方案 |

| 冲突示例 | `docs/conflict.md` | 用于制造和解决冲突的示例文件 |

| 总结报告 | `docs/05-summary-report.md` | 实验总结报告（全体成员共同完成） |

| 源代码 | `src/` | 系统源码（可选，本实验不强制开发） |

| 测试文件 | `test/` | 测试用例等（可选） |



\## 二、分支策略



本项目采用 Git Flow 简化版分支模型，包含以下分支：



\- \*\*`main`\*\*：主分支，受保护。只接受来自 `dev` 的合并，禁止直接推送。所有最终成果（包括总结报告）必须合并到 `main`。

\- \*\*`dev`\*\*：开发集成分支。所有功能分支（`feature/\*`）首先合并到 `dev`，经过集成验证后再合并到 `main`。

\- \*\*`feature/\*`\*\*：功能/文档分支。用于开发单个任务（如编写 WBS、进度计划、配置管理方案、制造冲突等）。命名示例：`feature/wbs`、`feature/config`、`feature/conflict-1`。



分支命名规范：

\- 使用小写字母和短横线，例如 `feature/update-wbs`

\- 禁止出现空格或特殊字符



\## 三、提交与合并规则



\### 3.1 提交规则

1\. 所有修改必须在自己的 `feature/\*` 分支上进行，不得直接在 `main` 或 `dev` 上提交。

2\. 提交信息格式：`<type>: <简短说明>`，其中 `<type>` 可以是：

&#x20;  - `docs`：文档更新（WBS、进度计划、配置管理等）

&#x20;  - `feat`：新功能

&#x20;  - `fix`：问题修复

&#x20;  - `chore`：杂项（如清理文件、配置调整）

&#x20;  - `conflict`：冲突解决相关

&#x20;  示例：`docs: 完成三级WBS分解`

3\. 提交应保持原子性：一次提交只做一件事。

4\. 每个提交必须包含真实的作者信息（`user.name` 设为本人姓名拼音全称，如 `liyouran`）。



\### 3.2 合并规则

1\. 功能分支开发完成后，由\*\*配置管理员\*\*（或指定审核人）负责审查并合并到 `dev` 分支。

2\. 合并前必须确保 `dev` 分支是最新的（`git pull origin dev`）。

3\. 合并时若产生冲突，由配置管理员联系相关成员共同解决，冲突解决后再次提交并继续合并。

4\. 合并到 `dev` 后，功能分支可以删除（可选）。

5\. 所有任务完成后，由组长或配置管理员将 `dev` 分支合并到 `main` 分支，并打版本标签（如 `v1.0`）。



\### 3.3 主分支保护规则

\- `main` 分支在 GitHub 上启用分支保护：

&#x20; - 要求 Pull Request 才能合并

&#x20; - 至少 1 人审核通过

&#x20; - 禁止强制推送

\- `dev` 分支建议也启用保护，但可允许配置管理员直接合并（根据小组约定）。



\## 四、冲突处理流程



1\. \*\*制造冲突\*\*：两个不同分支（如 `feature/conflict-a` 和 `feature/conflict-b`）修改同一文件的同一行，分别提交并先后合并到 `dev`。

2\. \*\*识别冲突\*\*：当合并第二个分支时，Git 会提示冲突，冲突文件会包含 `<<<<<<<`、`=======`、`>>>>>>>` 标记。

3\. \*\*解决冲突\*\*：

&#x20;  - 手动编辑冲突文件，删除标记，保留最终需要的内容。

&#x20;  - 使用 `git add <文件名>` 标记为已解决。

&#x20;  - 执行 `git commit` 完成合并提交。

4\. \*\*记录冲突\*\*：在总结报告（`docs/05-summary-report.md`）中说明冲突产生的原因、解决过程和最终结果。



\## 五、版本标记与发布



\- 每次将 `dev` 合并到 `main` 后，在 `main` 分支上打标签，格式为 `v<major>.<minor>.<patch>`，例如 `v1.0.0`。

\- 标签说明：

&#x20; - `v1.0.0`：实验最终成果（包含所有文档和冲突解决记录）



\## 六、日常协作约定



\- 每天开始工作前，先拉取最新的 `dev` 分支：`git checkout dev \&\& git pull origin dev`

\- 创建新分支前，确保 `dev` 是最新的。

\- 推送自己的分支后，在小组群内通知配置管理员或组长进行合并。

\- 实验报告由全体成员共同编辑，每个人补充自己负责的部分，最终合并到 `main`。
